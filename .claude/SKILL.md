# Налаштування Trello MCP

Підключає `@delorenj/mcp-server-trello` до Claude Code або Claude Desktop і доводить установку до стану «працює й перевірено».

## Головний принцип роботи

Користувач хоче, щоб ти виконував роботу сам, **але питав дозволу перед кожною зміною в його системі**. Це не формальність: конфіги MCP лежать поруч з іншими інтеграціями, і мовчазна перезапис чужого конфіга — найгірше, що тут може статися.

Тому:

- **Читати** файли, перевіряти версії, дивитися, що вже налаштовано — роби вільно, без питань.
- **Писати** файл, редагувати конфіг, запускати установку — покажи, що саме збираєшся зробити (шлях + точний diff чи команда), і дочекайся «так».
- Якщо користувач один раз сказав «роби далі без питань» — поважай це до кінця сесії.

Показуй diff, а не переказ. «Додам Trello в конфіг» — марно; користувач не може оцінити те, чого не бачить.

## Процес

### 0. Перевір, що ти взагалі там, де можеш це зробити

Цей skill змінює конфіг **на машині користувача**. Якщо ти працюєш у claude.ai чи іншому середовищі з тимчасовою пісочницею — ти не маєш доступу до тієї машини, і будь-яка твоя «установка» піде в контейнер, який зникне разом із сесією.

Це найнебезпечніший сценарій тут, бо він виглядає як успіх: команди відпрацюють, файли створяться, ти відрапортуєш «готово», а користувач нічого не отримає. Помилка виявиться через день і виглядатиме як магія.

Ознаки, що ти не на машині користувача:

- нема `~/.claude.json` чи конфігу Desktop там, де вони мають бути;
- `claude` не знаходиться як команда;
- домашня директорія порожня або належить `root`.

Перевір мовчки, до всього іншого:

```bash
which claude; ls ~/.claude.json 2>/dev/null || echo "конфігу нема"
```

Якщо це пісочниця — не імітуй установку. Скажи прямо, що вона можлива лише локально, віддай готові команди з підставленими значеннями і поясни, що далі робить користувач. Це корисна відповідь. Фальшивий рапорт про успіх — ні.

### 1. Розберися, з чого починаємо

Перш ніж щось пропонувати, з'ясуй поточний стан — це кілька секунд і рятує від дурних помилок:

- Який клієнт: Claude Code, Claude Desktop, чи обидва? Якщо неясно — спитай.
- Чи є вже Trello в конфізі? (можливо, він просто зламаний, і ставити наново не треба)
- Що є з рантаймів: `bun --version`, `node --version`. `bunx` — рекомендований шлях, але `npx` теж працює, тож відсутність Bun не є блокером.

### 2. Креденшели

Потрібні дві речі: `TRELLO_API_KEY` і `TRELLO_TOKEN`.

1. Ключ береться на <https://trello.com/app-key> (потребує Power-Up; сторінка проведе).
2. Токен генерується вручну **один раз на воркспейс** — відкрити в браузері:

```
https://trello.com/1/authorize?expiration=never&name=Claude%20Trello&scope=read,write&response_type=token&key=ВАШ_API_KEY
```

Підстав реальний ключ у `key=`. `name` — це те, що користувач побачить на екрані авторизації Trello, тож нехай буде впізнаваним. `scope=read,write` — мінімум для повноцінної роботи; якщо користувачу потрібен тільки перегляд, `scope=read` безпечніший.

Ти не можеш зробити цей крок за користувача — це браузерна авторизація. Дай готове посилання з підставленим ключем і почекай.

**Токен = повний доступ до бордів і `expiration=never`.** Поводься з ним відповідно: не друкуй у відповідях повністю, не клади у файли під git. Деталі — `references/security.md`, і прочитай його перед тим, як вибирати, куди писати креденшели.

### 3. Запис конфігу

Клієнти налаштовуються по-різному — прочитай потрібний референс, не покладайся на пам'ять:

- Claude Code → `references/claude-code.md`
- Claude Desktop → `references/claude-desktop.md`

Обидва — якщо користувач хоче Trello в обох місцях.

### 4. Перевірка

Установка без перевірки — це не установка. Конфіг може бути валідним JSON, а токен — простроченим, і користувач дізнається про це через тиждень посеред роботи.

Після перезапуску клієнта прогони по зростанню складності:

1. Чи з'явився сервер у списку MCP взагалі (`claude mcp list` або індикатор у Desktop) — перевіряє, що процес стартує.
2. `list_boards` — перевіряє, що ключ і токен справді валідні.
3. `get_active_board_info` — перевіряє стан, чи взагалі вибрано борд.

Якщо бордів багато, запропонуй одразу зафіксувати робочий: `set_active_board`. Вибір зберігається в `~/.trello-mcp/config.json` і переживає рестарти, тож це разова дія.

Не кажи «готово», поки не побачив реальний список бордів у відповіді.

### 5. Що тепер доступно

Покажи коротко, що з'явилось — інакше користувач має 30 інструментів і нуль уявлення, навіщо вони. Не переказуй увесь список: назви 4-5 речей, релевантних тому, що користувач розповів про свою роботу, і згадай, що повний перелік у `references/tools.md`.

## Коли щось не працює

`references/troubleshooting.md` — типові симптоми й що вони насправді означають. Читай його, коли сервер не піднявся, або інструменти не з'явились, або Trello повертає помилку.

Одна річ, яку варто знати наперед: **більшість «не працює» — це не помилка конфігу, а неперезапущений клієнт.** Перевір це першим.

## Про репозиторій

У `delorenj/mcp-server-trello` є власний skill-пакет (`skill/SKILL.md`) з інсталятором `skill/scripts/install.sh`, який збирає сервер локально з `skill/assets/source/`. Це альтернативний шлях — актуальний, якщо користувач хоче правити код сервера під себе. Для звичайного підключення `bunx @delorenj/mcp-server-trello` простіший і оновлюється сам.

## Чого не робити

- Не вигадуй синтаксис команд по пам'яті — CLI змінюється. `claude mcp add --help` коштує одну секунду.
- Не пропонуй `TRELLO_BOARD_ID`: він deprecated, `set_active_board` заміняє його і не потребує редагування конфігу.
- Не переписуй конфіг цілком, якщо треба додати один сервер — там уже можуть бути інші інтеграції.
- Не кажи «підключено», якщо не бачив реального списку бордів. Успішна команда ≠ робоча інтеграція.
